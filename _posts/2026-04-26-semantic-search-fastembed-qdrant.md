Traditional keyword search matches exact words. If a traveler asks for ***"quiet cafes near the Eiffel Tower"*** and our dataset contains ***"peaceful coffee shops with a view of the Iron Lady"***, keyword search returns nothing. Semantic search solves this by converting both the query and documents into mathematical vectors, dense numerical representations that capture meaning, not just tokens. It finds results that are conceptually related, even when the words don't overlap.

In this guide, we'll build a production-grade semantic search engine over a Paris subset of the Tourpedia dataset, specifically focusing on points of interest (POIs) categorized as attractions [Enriched Tourism Dataset Paris (POIs)](https://data.mendeley.com/datasets/vh4g4g2322/1).

We'll learn how to:

- Vectorize text using FastEmbed (Qdrant's lightweight, ONNX-powered embedding library).
- Store and index vectors with [Qdrant](https://qdrant.tech/) (the open-source vector database).
- Query semantically with cosine similarity, filtering, and hybrid search.
- Visualize and validate results using Qdrant's Web UI.

By the end, we'll have a search system that can answer ***"romantic sunset spots along the Seine"*** and surface relevant Parisian landmarks, without ever seeing those exact words in the data.

## 🐣 Semantic Search

Traditional keyword search behaves like a literal-minded genie: it scans an inverted index and only checks whether the exact wordss, or very close variations, appear in a document. So, if we ask it for "a **magical beast**" but our data contains "an **awesome dinosaur**", the genie fails to make the connection because it's only looking for those exact terms.

<img src="/images/posts/2026-04-26-semantic-search-fastembed-qdrant/1-genie-words-generated-gemini.png" width="90%" alt="Genie of words generated by Gemini" />

Semantic search takes a more intelligent approach. Instead of relying on exact word matches, it represents both queries and documents as vectors that capture meaning. It effectively asks, ***"Is this document conceptually similar to what the user is looking for?"*** By understanding context, synonyms, and intent, it can connect ideas that aren’t phrased the same way [[1](https://arxiv.org/html/2509.14457v1), [2](https://atlan.com/know/semantic-search-vs-keyword-search/#3-what-are-the-disadvantages-of-semantic-search), [3](https://www.couchbase.com/blog/semantic-search-vs-keyword-search-whats-the-difference/)]. In this sense, semantic search focuses on interpreting meaning rather than matching words, enabling it to deliver results that are more accurate and relevant.

**Examples of the Differences**

Let's illustrate the difference with a few real-world examples. Imagine we're searching a travel guide about France:

| User Query                                 | Keyword Search                                                                                                                                               | Semantic Search                                                                                                                                                                          |
| ------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| "Nice places to eat near the Eiffel Tower" | Looks for the exact words "Nice", "places", "eat", "Eiffel Tower". Might return results about the city of Nice, or miss results that say "dining" or "cafe". | Understands the intent: Find restaurants in proximity to a landmark. It will return pages about "dining", "restaurants", "cafes", and "brasseries" near the Eiffel Tower.                |
| "A place for a romantic evening"           | Looks for "romantic" and "evening". It might miss a page that says "perfect for anniversaries with a sunset view" because the exact keywords are missing.    | Recognizes "romantic evening" is conceptually similar to "anniversary dinner" or "sunset date". It will surface those pages based on meaning.                                            |
| "How to get from CDG to the Louvre"        | Looks for "CDG", "Louvre", "get". It will match the words perfectly, but a typo like "how to go from CDG to the Luvre" would fail.                           | Identifies the entities and their relationship. It will understand "CDG" is Charles de Gaulle Airport and "Louvre" is a museum, and that the query is about transportation between them. |

**Beyond the Hype**

Semantic search isn’t always the clear winner. In fact, the traditional keyword-based algorithm BM25, which doesn’t rely on training data and is purely statistical, still outperforms dense retrieval on 9 of the 18 datasets in the BEIR benchmark under zero-shot conditions [[4]](https://arxiv.org/pdf/2104.08663). This highlights that classical methods remain valuable, especially for precise matches such as codes, IDs, or jargon-heavy queries.

## 🧘 Vectorization

Vectorization is the process of transforming text, image, or audio, into numerical representations called vectors. A well-designed vector captures the semantic meaning of the data, placing similar concepts close to one another in a high-dimensional space.

**Types of Vectorization**

- Dense Embeddings: These produce a single, compact vector (e.g., 384, 768, or 1536 dimensions) that represents the overall meaning of the input as a point in a high-dimensional space [[5]](https://www.elastic.co/search-labs/jp/blog/sparse-vector-embedding).
- Sparse Embeddings: These are high-dimensional vectors where most values are zero [[5](https://www.elastic.co/search-labs/jp/blog/sparse-vector-embedding), [6](https://docs.opensearch.org/2.14/search-plugins/neural-sparse-search/)]. They excel at capturing specific keywords and exact term matching within a semantic space, making them ideal for hybrid search [[7]](https://link.springer.com/chapter/10.1007/978-3-540-68234-9_41).
- Multimodal Embeddings: These models can process different types of data, such as text, images, audio, or video, and map them into a shared vector space, enabling cross-modal understanding [[8]](https://www.alibabacloud.com/help/en/model-studio/embedding).
- Multi-Vector Embeddings (Late Interaction): Models like ColBERT generate multiple vectors per document, allowing for token-level matching and finer-grained retrieval [[9]](https://qdrant.tech/course/essentials/day-5/colbert-multivectors/).

## 🔍 Similarity Metrics

Once we have our query vector and a database of document vectors, we need a way to find which ones are nearest to each other. This is where similarity metrics come in. Qdrant supports four main [distance metrics](https://www.mintlify.com/qdrant/qdrant/concepts/distance-metrics) out of the box [[10]](https://www.mintlify.com/qdrant/qdrant/concepts/distance-metrics).

### 1. Cosine Similarity

Measures the cosine of the angle between two vectors, ignoring their magnitude. It essentially cares about their direction.

- Score Range: -1 (opposite directions) to 1 (identical direction).
- When to Use: The most common metric for semantic search and text embeddings. Most pre-trained models (OpenAI, Cohere, Sentence Transformers) are optimized for cosine similarity.

### 2. Euclidean Distance (L2)

Measures the straight-line distance between the ends of two vectors.

- Score Range: 0 (identical) to ∞ (very different).
- When to Use: When vector magnitude is meaningful, commonly for image embeddings (ResNet, EfficientNet).

### 3. Dot Product

Calculates the product of corresponding elements in two vectors.

- Score Range: -∞ to ∞.
- When to Use: Most efficient metric, especially on GPUs. If our vectors are normalized to unit length, the dot product is equivalent to cosine similarity and is faster to compute.

> ⚠️ Dot product is sensitive to vector magnitude. Longer vectors will have higher scores even if less similar in direction.

### 4. Manhattan Distance

Measures the distance along axes at right angles. It's less common in vector search but useful in certain high-dimensional spaces.

| Metric                | Best For                               | Type                          |
| --------------------- | -------------------------------------- | ----------------------------- |
| **Cosine**      | L2-normalized vectors, text embeddings | Similarity (higher is better) |
| **Euclidean**   | Raw vectors with meaningful norms      | Distance (lower is better)    |
| **Dot Product** | Transformer embeddings, GPU efficiency | Similarity (higher is better) |

## 🚀 State-of-the-Art in Semantic Search (2025–2026)

Semantic search isn't standing still. So much has happened recently in this space, reshaping how we think about information retrieval. Here's what's current in 2026.

### 📊 What the Benchmarks Tell Us

MTEB (Massive Text Embedding Benchmark) has become the standard for evaluating embedding models. Based on the latest 2026 data from the [Hugging Face MTEB leaderboard](https://huggingface.co/spaces/mteb/leaderboard), the top models for retrieval performance include Microsoft's Harrier-OSS-v1 model series, KaLM-Embedding-Gemma3-12B-2511, Qwen3 Embedding (8B), and Gemini Embeddings 2 [[11]](https://huggingface.co/spaces/mteb/leaderboard).

> ⚠️ The leaderboard changes constantly with frequent new submissions, so it's best to double-check scores before we choose a model.

### 🏆 Hybrid Retrieval is Now the Baseline

The old debate of "keyword vs. semantic" is dead. At [ECIR 2026](https://ecir2026.eu/), *every* production search system presented used  **hybrid retrieval**, combining keyword methods like BM25 with dense semantic vectors [[12](https://github.com/DataTalksClub/llm-zoomcamp/blob/main/04-monitoring/chat_starter_qdrant/documents_extraction.py), [13](https://www.algolia.com/blog/engineering/ecir-2026-recap)]. The question has shifted from *"should we?"* to *"how efficiently and at what cost?"*.

### 🤹 Multi-Vector & Late Interaction Models

Multi-vector models like **ColBERT** are gaining attention [[14]](https://arxiv.org/html/2603.25248v1). They create multiple embeddings per document to enable token-level matching, providing a nuanced middle ground between sparse and dense retrieval.

### ✨ Multimodal Search

The era of search across different media is here. Libraries include [**Gemini Embedding 2**](https://docs.cloud.google.com/vertex-ai/generative-ai/docs/models/gemini/embedding-2), **LanguageBind** [[15]](https://arxiv.org/pdf/2310.01852), and [**UForm**](https://github.com/unum-cloud/uform) can all map text, images, and video into a shared vector space, enabling cross-modal retrieval.

**☀️ Extract Review Text and Metadata**
The *reviews* column is a potential source of critical data, so we will now extract that information.

> *Function to extract review text + metadata*

```python
# --- Function to extract review text + metadata ---
import ast
def extract_reviews(data):
    if pd.isna(data):
        return "", [], []
  
    try:
        parsed = ast.literal_eval(data)  # convert string → Python object
      
        if isinstance(parsed, list):
            texts = []
            ratings = []
            polarities = []
          
            for item in parsed:
                if isinstance(item, dict):
                    # Extract fields safely
                    text = item.get("text", "")
                    rating = item.get("rating", None)
                    polarity = item.get("polarity", None)
                  
                    if text:
                        texts.append(text)
                    if rating is not None:
                        ratings.append(rating)
                    if polarity is not None:
                        polarities.append(polarity)
          
            # Combine all review texts into one string
            combined_text = " ".join(texts)
          
            return combined_text, ratings, polarities
  
    except Exception:
        pass

    # fallback if parsing fails
    return str(data), [], []
```

> *Apply extraction*

```python
# Apply extraction
df[["clean_reviews", "ratings", "polarities"]] = df["reviews"].apply(
    lambda x: pd.Series(extract_reviews(x))
)

# Keep only useful column for IR
reviews_df = df[["clean_reviews"]]

print(reviews_df.head())

```

### 📙 Part 2: Embedding, Indexing, and Basic Search (Core pipeline)

#### ☑️ Embedding Model Selection

Now we know that we’re working with English text, the next step is to select an appropriate embedding model to convert this content into vector representations.

Choosing the right embedding model depends on several factors, including the task, the type of data, and its specific characteristics. We also need to consider the trade-off between retrieval quality and resource usage, since larger embeddings typically require more storage and memory. In addition, inference cost can become important, especially when relying on third-party APIs.

In practice, the most reliable approach is to experiment with multiple models and evaluate their performance on our own data.

In this guide, we will use FastEmbed as the embedding provider. FastEmbed is an embedding solution tailored for use with Qdrant. It is designed for low-latency, CPU-efficient embedding generation.

#### 🗂️ Creating a Qdrant Collection

Before indexing, we need a  **collection** to store our vectors (it's like a table in a database). To create a collection in Qdrant, the essential configuration requires:

* **Name:** A unique identifier for the collection.
* **Vector Configuration:**
  * **Size:** The dimensionality of the embedding vectors.
  * **Distance Metric:** The function used to calculate similarity between vectors (e.g., Cosine, Euclidean).

While Qdrant supports advanced [configuration](https://qdrant.tech/documentation/manage-data/collections/#create-a-collection), including alternative vector types for use cases like hybrid search, this example uses a simple default setup suitable for typical dense embeddings.

```python
from qdrant_client import QdrantClient
from qdrant_client.models import Distance, VectorParams

client = QdrantClient(host="localhost", port=6333)

collection_name = "french_tourism"

# Check if collection exists
collections = client.get_collections().collections
collection_names = [c.name for c in collections]

if collection_name not in collection_names:
    client.create_collection(
        collection_name=collection_name,
        vectors_config=VectorParams(
            size=384,
            distance=Distance.COSINE
        )
    )
    print(f"Collection '{collection_name}' created ✅")
else:
    print(f"Collection '{collection_name}' already exists 👍")
```

#### 📝 Creating Data Points and Indexing

Now we'll convert our text reviews into embeddings using FastEmbed and store them as "points" in Qdrant.

***Generate Embeddings using the Embedding Model***

```python
from fastembed import TextEmbedding
from qdrant_client import models

# Initialize the embedding model
embedding_model = TextEmbedding(model_name="BAAI/bge-small-en-v1.5")

# Prepare documents: combine destination name + review text for richer context
documents = [
    f"Destination: {row['name']}, Address: {row['address']}. Review: {row['clean_reviews']}"
    for _, row in df.iterrows()
]

# Generate embeddings in batches
print("Generating embeddings...")
embeddings = list(embedding_model.embed(documents))
```

***Create Payloads***

```python
# Create payloads (for metadata filtering)

payloads = [
    {
        "destination_id": str(row["id"]),
        "category": row["subCategory"],
        "destination_name": row["name"],
        "address": row["address"],
        "latitude": row["lat"],
        "longitude": row["lng"],
        "rating": row["polarities"],
        "review_text": row["clean_reviews"]
    }
    for _, row in df.iterrows()
]
payloads

```

***Create Points***
Points are the core data entities in Qdrant. Each point consists of:

- ID. A unique identifier. Qdrant supports both 64-bit unsigned integers and UUIDs.
- Vector. The embedding that represents the data point in vector space.
- Payload (optional). Additional metadata as key-value pairs.

```python
# Create points 

points = [
    models.PointStruct(
        id=idx,
        vector=embedding.tolist(),
        payload=payload
    )
    for idx, (embedding, payload) in enumerate(zip(embeddings, payloads))
]
```

***Upsert Generated Points into the Collection***

```python
# Upsert generated points into the collection
client.upsert(
    collection_name=collection_name,
    points=points
)

print(f"Indexed {len(points)} points successfully!")
```

#### 🔡 Semantic Search 
Let's validate our semantic search system with both known and novel queries.

```python
def semantic_search(query, limit=5, filter_category=None):
    """
    Perform semantic search with optional filtering
    """
    query_embedding = next(embedding_model.embed([query]))

    # Build filter if needed
    search_filter = None
    if filter_category:
        search_filter = models.Filter(
            must=[
                models.FieldCondition(
                    key="category",
                    match=models.MatchValue(value=filter_category)
                )
            ]
        )

    results = client.query_points(
        collection_name=collection_name,
        query=query_embedding.tolist(),
        limit=limit,
        query_filter=search_filter
    ).points

    return results

# Test with known concepts
queries = [
    "Where can I find beautiful castles?",
    "Top rated restaurants in coastal towns",
    "Family-friendly hiking trails",
    "Places with amazing sunset views",
    "Romantic getaways for couples"
]

for q in queries:
    print(f"\n🔍 Query: {q}")
    results = semantic_search(q, limit=3)
    for i, r in enumerate(results, 1):
        print(f"  {i}. {r.payload['destination_name']} (score: {r.score:.3f})")
```

## 🏁 Conclusion

As we wrap up our journey from keyword matching to meaning‑driven discovery, here are the key takeaways from building a semantic search engine over Paris tourism data with FastEmbed and Qdrant:

* ✅ **Semantic search bridges the gap between human intent and exact keywords**

  - By converting text into dense vectors with FastEmbed, we transformed a raw Paris tourism dataset into an intelligent discovery engine.
* ✅ **FastEmbed + Qdrant provide a production-ready, lightweight stack**

  - FastEmbed's ONNX-powered inference delivers speed and low dependencies, while Qdrant handles billions of vectors with cosine similarity, filtering, and hybrid search; all accessible via a clean Python API and a visual Web UI.
* ✅ **Real data makes the difference**

  - Using the Mendeley French tourism dataset, we built, indexed, and queried a semantic search system in under 100 lines of code. The results speak for themselves: queries return meaning‑matched destinations, not just token overlaps.
* ✅ **State‑of‑the‑art is within reach**

  - With models like Microsoft Harrier, Qwen3, and Gemini Embedding 2 topping the MTEB leaderboard, and hybrid retrieval becoming the baseline, we can continuously upgrade our search quality without rewriting our infrastructure.
* ✅ **Our next step: go multimodal**

  - Extend this blog's case study by adding image embeddings (CLIP/LanguageBind) or audio/video search. Qdrant's multi‑vector and payload filtering make it easy to scale from text-only to rich, cross‑modal discovery.
