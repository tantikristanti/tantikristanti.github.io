If a traveler asks for ***"quiet cafes near the Eiffel Tower"*** and our dataset contains ***"peaceful coffee shops with a view of the Iron Lady"***, keyword search returns nothing. Semantic search solves this by converting both the query and documents into mathematical vectors, dense numerical representations that capture meaning, not just tokens. - Visualize and validate results using Qdrant's Web UI.

## 🐣 Semantic Search

Traditional keyword search behaves like a literal-minded genie: it scans an inverted index and only checks whether the exact words, or very close variations, appear in a document. So, if we ask it for "a **magical beast**" but our data contains "an **awesome dinosaur**", the genie fails to make the connection because it's only looking for those exact terms.

<img src="/images/posts/2026-04-26-semantic-search-fastembed-qdrant/1-genie-words-generated-gemini.png" width="90%" alt="Genie of words generated by Gemini" />

Semantic search takes a more intelligent approach. Instead of relying on exact word matches, it represents both queries and documents as vectors that capture meaning. It effectively asks, ***"Is this document conceptually similar to what the user is looking for?"*** By understanding context, synonyms, and intent, it can connect ideas that aren’t phrased the same way (1-3). In this sense, semantic search focuses on interpreting meaning rather than matching words, enabling it to deliver results that are more accurate and relevant.

**Examples of the Differences**

Let's illustrate the difference with a few real-world examples. Imagine we're searching a travel guide about France:

| User Query                                 | Keyword Search                                                                                                                                               | Semantic Search                                                                                                                                                                          |
| ------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| "Nice places to eat near the Eiffel Tower" | Looks for the exact words "Nice", "places", "eat", "Eiffel Tower". Might return results about the city of Nice, or miss results that say "dining" or "cafe". | Understands the intent: Find restaurants in proximity to a landmark. It will return pages about "dining", "restaurants", "cafes", and "brasseries" near the Eiffel Tower.                |
| "A place for a romantic evening"           | Looks for "romantic" and "evening". It might miss a page that says "perfect for anniversaries with a sunset view" because the exact keywords are missing.    | Recognizes "romantic evening" is conceptually similar to "anniversary dinner" or "sunset date". It will surface those pages based on meaning.                                            |
| "How to get from CDG to the Louvre"        | Looks for "CDG", "Louvre", "get". It will match the words perfectly, but a typo like "how to go from CDG to the Luvre" would fail.                           | Identifies the entities and their relationship. It will understand "CDG" is Charles de Gaulle Airport and "Louvre" is a museum, and that the query is about transportation between them. |

**Beyond the Hype**

Semantic search isn’t always the clear winner. In fact, the traditional keyword-based algorithm BM25, which doesn’t rely on training data and is purely statistical, still outperforms dense retrieval on 9 of the 18 datasets in the BEIR benchmark under zero-shot conditions (4). This highlights that classical methods remain valuable, especially for precise matches such as codes, IDs, or jargon-heavy queries.

## 🧘 Vectorization

Vectorization is the process of transforming text, image, or audio, into numerical representations called vectors. A well-designed vector captures the semantic meaning of the data, placing similar concepts close to one another in a high-dimensional space.

**Types of Vectorization**

- Dense Embeddings: These produce a single, compact vector (e.g., 384, 768, or 1536 dimensions) that represents the overall meaning of the input as a point in a high-dimensional space (5).
- Sparse Embeddings: These are high-dimensional vectors where most values are zero (5,6). They excel at capturing specific keywords and exact term matching within a semantic space, making them ideal for hybrid search (7).
- Multimodal Embeddings: These models can process different types of data, such as text, images, audio, or video, and map them into a shared vector space, enabling cross-modal understanding (8).
- Multi-Vector Embeddings (Late Interaction): Models like ColBERT generate multiple vectors per document, allowing for token-level matching and finer-grained retrieval (9).

## 🔍 Similarity Metrics

Once we have our query vector and a database of document vectors, we need a way to find which ones are nearest to each other. This is where similarity metrics come in. Qdrant supports four main [distance metrics](https://www.mintlify.com/qdrant/qdrant/concepts/distance-metrics) out of the box (10).

### 1. Cosine Similarity

Measures the cosine of the angle between two vectors, ignoring their magnitude. It essentially cares about their direction.

- Score Range: -1 (opposite directions) to 1 (identical direction).
- When to Use: The most common metric for semantic search and text embeddings. Most pre-trained models (OpenAI, Cohere, Sentence Transformers) are optimized for cosine similarity.

### 2. Euclidean Distance (L2)

Measures the straight-line distance between the ends of two vectors.

- Score Range: 0 (identical) to ∞ (very different).
- When to Use: When vector magnitude is meaningful, commonly for image embeddings (ResNet, EfficientNet).

### 3. Dot Product

Calculates the product of corresponding elements in two vectors.

- Score Range: -∞ to ∞.
- When to Use: Most efficient metric, especially on GPUs. If our vectors are normalized to unit length, the dot product is equivalent to cosine similarity and is faster to compute.

> ⚠️ Dot product is sensitive to vector magnitude. Longer vectors will have higher scores even if less similar in direction.

### 4. Manhattan Distance

Measures the distance along axes at right angles. It's less common in vector search but useful in certain high-dimensional spaces.

| Metric                | Best For                               | Type                          |
| --------------------- | -------------------------------------- | ----------------------------- |
| **Cosine**      | L2-normalized vectors, text embeddings | Similarity (higher is better) |
| **Euclidean**   | Raw vectors with meaningful norms      | Distance (lower is better)    |
| **Dot Product** | Transformer embeddings, GPU efficiency | Similarity (higher is better) |

#### ☀️ Step 7: Extract Review Texts and Metadata

The *reviews* column is a potential source of critical data, so we will now extract that information.

> *Function to extract review text + metadata*

```python
# --- Function to extract review text + metadata ---
def extract_reviews(data):
    if pd.isna(data):
        return "", [], []
  
    try:
        parsed = ast.literal_eval(data)  # convert string → Python object
  
        if isinstance(parsed, list):
            texts = []
            ratings = []
            polarities = []
    
            for item in parsed:
                if isinstance(item, dict):
                    # Extract fields safely
                    text = item.get("text", "")
                    rating = item.get("rating", None)
                    polarity = item.get("polarity", None)
            
                    if text:
                        texts.append(text)
                    if rating is not None:
                        ratings.append(rating)
                    if polarity is not None:
                        polarities.append(polarity)
    
            # Combine all review texts into one string
            combined_text = " ".join(texts)
    
            return combined_text, ratings, polarities
  
    except Exception:
        pass

    # fallback if parsing fails
    return str(data), [], []

```

#### ☑️ Step 8: Choose Embedding Model

Now we know that we’re working with English text, the next step is to select an appropriate embedding model to convert this content into vector representations (24).

Choosing the right embedding model depends on several factors, including the task, the type of data, and its specific characteristics. We also need to consider the trade-off between retrieval quality and resource usage, since larger embeddings typically require more storage and memory. In addition, inference cost can become important, especially when relying on third-party APIs.

In practice, the most reliable approach is to experiment with multiple models and evaluate their performance on our own data.

In this guide, we will use FastEmbed as the embedding provider. FastEmbed is an embedding solution tailored for use with Qdrant. It is designed for low-latency, CPU-efficient embedding generation.

#### 🗂️ Step 9: Create a Qdrant Collection

Before indexing, we need a  **collection** to store our vectors (it's like a table in a database). To create a collection in Qdrant, the essential configuration requires:

* **Name:** A unique identifier for the collection.
* **Vector Configuration:**
  * **Size:** The dimensionality of the embedding vectors.
  * **Distance Metric:** The function used to calculate similarity between vectors (e.g., Cosine, Euclidean).

While Qdrant supports advanced [configuration](https://qdrant.tech/documentation/manage-data/collections/#create-a-collection), including alternative vector types for use cases like hybrid search, this example uses a simple default setup suitable for typical dense embeddings.

```python
collection_name = "french_tourism"

# Check if collection exists
collections = client.get_collections().collections
collection_names = [c.name for c in collections]

if collection_name not in collection_names:
    client.create_collection(
        collection_name=collection_name,
        vectors_config=VectorParams(
            size=384,
            distance=Distance.COSINE
        )
    )
    print(f"Collection '{collection_name}' created ✅")
else:
    print(f"Collection '{collection_name}' already exists 👍")

```

#### 📝 Step 10: Generate Embeddings & Insert Points

> ***Generate Embeddings***

We'll vectorize the `name`, `address`, and `clean_reviews` columns into embeddings using FastEmbed and store them as "points" in Qdrant. The aim is to embed information about a place and also reviews from tourists about a particular tourist point of interest.

```python
# Prepare documents: combine destination name + review text for richer context

    for _, row in df.iterrows()

# Generate embeddings in batches

```

> ***Prepare Payloads***

In addition, we will use the `subCategory`, `name`, `address`, `lat`, `lng`, `polarities`, `reviews`, and `avg_polarity` columns as metadata. This allows us to apply filters to tourist points of interest, for example, finding pubs near a hotel that have strong ratings.

```python
# Create payloads (metadata for filtering)
payloads = [
    {
        "subCategory": row["subCategory"],
        "name": row["name"],
        "address": row["address"],
        "lat": row["lat"],
        "lng": row["lng"],
        "polarities": row["polarities"],
        "reviews": row["clean_reviews"],
        "avg_polarity": row["avg_polarity"],
    }
    for _, row in df.iterrows()
]

```

> ***Create Points***

Points are the core data entities in Qdrant. Each point consists of:

- ID. A unique identifier. Qdrant supports both 64-bit unsigned integers and UUIDs.
- Vector. The embedding that represents the data point in vector space.
- Payload (optional). Additional metadata as key-value pairs.

```python
# Create points 
points = [
    models.PointStruct(
        id=idx,
        vector=embedding.tolist(),
        payload=payload
    )
    for idx, (embedding, payload) in enumerate(zip(embeddings, payloads))
]

```

***Insert Generated Points into the Collection***

For production, use `upload_points()` for large datasets. Here we use `upsert()` for simplicity.

```python
# Upsert generated points into the collection
client.upsert(
    collection_name=collection_name,
    points=points
)

print(f"Indexed {len(points)} points successfully!")
```

### 🧪 Part 4: Similarity Search and Visualization

#### 🆎 Step 12: Semantic Search without Any Payload Filtering

Let's perform vector search without any payload filtering using the query ***"peaceful coffee shops with a view of the Eiffel Tower"***.

```python
def semantic_search(query, limit=5, filter_category=None):
    """
    Perform semantic search without any payload filtering
    Args:
        query (str): User's natural language query
        limit (int): Number of results to return
    Returns:
        list of ScoredPoint objects with score and payload
    """

    # 1. Convert query to embedding (generator -> next() gives first vector)
    query_embedding = next(embedding_model.embed([query])) 
  
    # 2. Search in Qdrant
    results = client.query_points(
        collection_name=collection_name,
        query=query_embedding.tolist(),  # Pass vector directly
        limit=limit,
        with_payload=True  # Include metadata in results
    ).points

    return results
```

#### 🔡 Step 13: Semantic Search with Payload Filtering

Filtering allows narrowing results based on metadata. Common filter types:

- `MatchValue`: exact match on a string or number.
- `Range`:  numeric/date range (e.g., `gte`, `lte`).
- `MatchAny`: list of possible values.

> **Note:** Since `polarities` is a list, we use `avg_polarity` (scalar) for rating-based filtering.

Let's validate our semantic search system and test it with both known and novel queries.

```python
def semantic_search_with_filters(query, limit=5, min_rating=None, category=None):
    # Vector search with optional filters on avg_polarity and subCategory.
    query_embedding = next(embedding_model.embed([query]))
    # Build filter conditions
    must_conditions = []
    if min_rating is not None:
        must_conditions.append(models.FieldCondition(
                                key="avg_polarity",
                                range=models.Range(gte=min_rating))) # gte: greater than or equal ( >= )
    if category is not None:
        must_conditions.append(models.FieldCondition(
                                key="subCategory",
                                match=models.MatchValue(value=category)))
    search_filter = models.Filter(must=must_conditions) if must_conditions else None
    results = client.query_points(collection_name=collection_name,
                                query=query_embedding.tolist(),
                                limit=limit,          
                                query_filter=search_filter,
                                with_payload=True
                                ).points
    return results
```

#### 📸 Step 15: Best Practices & Next Steps

***Performance Tips***

- **Batch upload**: For >10k points, use `client.upload_points()` (handles lazy batching, retries, parallelism).
- **Index payload fields**: If you filter frequently (e.g., `subCategory`), create a payload index to speed up queries.
- **Limit search results**: Keep `limit` between 10–100; use `score_threshold` to exclude low-quality matches.

 ***Potential Improvements for This Dataset***

- **Geographic search**: Qdrant supports geo-radius filtering (`geo_radius` on `lat`/`lng`) payload.
- **Hybrid search**: Combine dense (semantic) + sparse (BM25) for better keyword matching on proper nouns.
- **Reranking**: Use a cross-encoder to reorder top-k results for higher precision.
